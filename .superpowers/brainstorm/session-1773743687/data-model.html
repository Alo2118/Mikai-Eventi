Sezione 1: Modello Dati

Le tabelle principali e le relazioni tra di loro

Schema Database — Entità Principali

👤 users

id, email, nome, cognome
ruolo: admin | ufficio | commerciale
telefono, avatar_url
attivo: si/no

📅 events

id, titolo, tipo_evento
(workshop|corso|congresso|convegno|cadlab|live_surgery)
luogo, data_inizio, data_fine
desk_richiesto: si/no
stato: proposto → confermato → in_preparazione → in_corso → concluso
note, contributo_economico_only
promotore_id → users
created_by → users

📦 materials

id, nome, tipo
(demo_kit|montaggio|locandina|depliant|video|vela|altro)
codice_inventario, quantita_totale
posizione_attuale: magazzino | evento | agente | cliente | spedito | manutenzione
posizione_dettaglio (testo libero)
foto_url, note

🔄 material_movements

id, material_id → materials
event_id → events (opzionale)
tipo: uscita | rientro | trasferimento
da_posizione, a_posizione
data_movimento, data_rientro_prevista
responsabile_id → users
tracking_spedizione, note

👥 event_staff

id, event_id → events
user_id → users
ruolo_evento: (responsabile|staff|relatore|ospite)
confermato: si/no

📋 event_materials

id, event_id → events
material_id → materials
quantita_richiesta
stato: richiesto → approvato → preparato → spedito → consegnato → rientrato
richiesto_da → users
data_richiesta, note

📇 event_participants

id, event_id → events
nome, cognome, email
ente_ospedaliero, ruolo_medico
stato_iscrizione: invitato → confermato → presente → assente
note

⏰ event_deadlines

id, event_id → events
tipo: (dati_evento|materiale_marketing|iscrizioni|logistica)
descrizione, data_scadenza
assegnato_a → users
completato: si/no, completato_il
notifica_inviata: si/no

Relazioni chiave

📅 Evento → ha molti 👥 Staff, 📋 Materiali richiesti, 📇 Partecipanti, ⏰ Scadenze
📦 Materiale → ha molti 🔄 Movimenti (storico completo di dove è stato)
👤 Utente → può essere promotore di eventi, staff, responsabile movimenti
🔄 Movimento → collega un materiale a un evento con date uscita/rientro
